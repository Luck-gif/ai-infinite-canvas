### 1.4 LibTV 真实页面验证与市场定位

#### 1.4.1 调研结论

2026-07-19 使用 Playwright 对 `https://www.liblib.tv/detail/89d84659abc04965bb1872a1d966400d` 进行真实页面渲染抓取，验证如下：

| 维度 | 发现 | 对本项目的映射 |
|------|------|----------------|
| **产品形态** | LibTV 定位为「专业视频创作工具」；作品详情页展示创作者、更新时间、含 AI 生成内容声明 | 本项目同样定位「自用开发、保留产品化能力」的专业创作工具，方向一致 |
| **创作过程可展示** | 提供「查看制作过程」按钮，弹出 **工作流 / 故事板** 双 Tab 对话框 | 本项目已落地 WorkflowGraph（§6.4）+ WorkflowPanel（§6.4.3），可继续补齐故事板视图 |
| **工作流视图** | 以 DAG 画布展示「图片节点」与「视频节点」的连接关系，支持小地图、节点连线、缩放 | 与项目当前 Canvas/WorkflowGraph 设计一致，但 LibTV 节点规模更大、参考素材更完整 |
| **故事板视图** | 以列表/序列形式展示每个镜头节点，标注：节点类型、模型名、时长、分辨率、参考素材 | 对应项目规划的 StoryboardNode（§6.4.1），但尚未实现时间线/列表形态的故事板 |
| **参考素材** | 每个视频节点挂载「参考素材」元素，可「查看该参考元素详情」 | 对应项目规划的 BlendNode / 一致性控制 / IP-Adapter 参考图，目前尚未落地参考素材节点 |
| **复制项目** | 只读模式下提供「复制项目」入口，允许用户基于该工作流二次创作 | 可作为本项目「用户自定义工作流保存/加载 + GPT 辅助创建」的竞品参照 |
| **典型节点示例** | 视频节点：`视频节点 41 star-video2 15秒 1280 x 720` | 当前项目视频节点已支持 `frames/fps`，但尚未显示模型名、分辨率、时长等元数据 |

> **核心验证**：LibTV 的「工作流 + 故事板」双视图、「图片节点 + 视频节点 + 参考素材节点」三类型节点、以及「复制项目」二次创作模式，均与项目 v4.5+ 规划高度一致，确认分镜编排、多图融合、一致性控制等方向具备真实市场需求。

#### 1.4.2 市场定位与差异化

| 竞品 | 定位 | 画布形态 | 推理后端 | 关键差异 |
|------|------|---------|---------|----------|
| **LibTV** | 国产云端专业视频创作 SaaS | 工作流+故事板双视图 | 闭源云端模型 | 云端、闭源、需联网；本项目本地优先、隐私可控 |
| **basketikun/infinite-canvas** | 开源无限画布工作台 | 节点式画布 | OpenAI 兼容 API | AGPL-3.0 强 Copyleft；本项目全栈 MIT/Apache 更产品化 |
| **Invoke AI** | 艺术家开源创意引擎 | Unified Canvas + 节点工作流 | 自带 SD 引擎 | 自研引擎；本项目仅 API 调 ComfyUI 规避 GPL 传染 |
| **ComfyUI** | 节点 DAG 推理引擎 | 自带前端节点画布 | 自身 | GPL-3.0；本项目仅 API 调用，不 fork/分发 |
| **本项目** | 本地优先、AI 驱动、保留产品化 | Konva 自研节点画布 | 原生 ComfyUI（仅 API） | 独特卖点：自然语言 → 意图 → 工作流自动生成 |

> **市场信号**：OpenAI Sora 于 2026-03 关停消费级 App/API，印证纯云端高算力模式商业化困难，本地优先/开源/混合是 2026 年明确趋势。本项目「本地优先 + ComfyUI 仅 API + 意图驱动工作流自动生成」构成差异化竞争力。

#### 1.4.3 对标学习项（建议纳入 v4.28+）

1. **故事板列表视图**：在现有 WorkflowGraph 之外，增加时间线/列表形式的故事板视图，展示每个镜头的模型、时长、分辨率、参考素材。
2. **参考素材节点**：将 IP-Adapter / LoRA / ControlNet / 参考图显式化为可拖拽、可查看详情的「参考素材」节点。
3. **工作流复制/克隆**：支持用户保存当前画布为模板，并基于他人/历史工作流二次创作。
4. **节点元数据 richer**：视频节点显示模型名、时长、分辨率；图片节点显示生成模型、seed、模板等。

## 2. 开发原则（AI 驱动）

### 2.1 AI 协作开发流程

```
需求明确 → AI 生成代码 → 人工审查 → 自动化测试 → 修复迭代 → 集成部署
    ↑                                                        ↓
    └──────────────── 反馈循环 ←──────────────────────────────┘
```

### 2.2 开发原则

| 原则 | 说明 |
|------|------|
| **先设计后编码** | 每次功能开发前，先输出设计文档（接口定义/数据流/测试用例），经确认后再生成代码 |
| **小步快跑** | 每个功能拆分为可独立测试的最小单元，逐个实现验证 |
| **测试先行** | 先写测试用例，再写实现代码，确保测试覆盖率 ≥ 80% |
| **文档同步** | 代码变更同步更新文档（API 文档/架构文档/CHANGELOG） |
| **合规检查** | 每次提交前检查 GPL 合规性（不引入 ComfyUI 代码） |
| **版本锁定** | 所有依赖锁定版本，避免隐式升级导致的不兼容 |

### 2.3 AI 生成代码要求

| 要求 | 说明 |
|------|------|
| **类型安全** | TypeScript 严格模式，禁止 `any`（除非必要且注释说明） |
| **注释规范** | 公共接口必须有 JSDoc/Docstring，说明参数/返回值/异常 |
| **错误处理** | 所有外部调用（API/WS/文件系统）必须有 try-catch 和降级策略 |
| **日志记录** | 关键操作记录日志（info/warn/error），便于调试 |
| **可测试性** | 纯函数优先，依赖注入，避免全局状态 |

```typescript
function motionToParams(motion: number, model: string): Record<string, number> {
  switch (model) {
    case 'wan22_i2v':
      return { motion_bucket_id: Math.round(motion * 2.55) };  // 0-255
    case 'ltx23_i2v':
      return { motion_strength: motion / 100 };  // 0.0-1.0
    case 'cogvideox':
      return { guidance_scale: 1.0 + (motion / 100) * 9 };  // 1.0-10.0
    default:
      return {};
  }
}
```
